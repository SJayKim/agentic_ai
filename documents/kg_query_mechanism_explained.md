# Knowledge Graph 쿼리 메커니즘 — 쉬운 설명

> 이 문서는 LightRAG 기반 Knowledge Graph 쿼리의 핵심 단계(3-1 ~ 3-4)를
> 처음 보는 사람도 이해할 수 있도록 비유와 예시를 들어 설명합니다.

---

## 비유: 도서관에서 책 찾기

KG 쿼리를 **거대한 도서관에서 원하는 정보를 찾는 과정**에 비유하겠습니다.

- **엔티티(Entity)** = 도서관의 **인물 카드, 회사 카드, 프로젝트 카드** (색인 카드)
- **관계(Relationship)** = 카드와 카드를 잇는 **실(연결선)**. "A가 B에 소속", "C가 D를 개발" 같은 것
- **텍스트 청크** = 실제 **책의 페이지** (원본 내용)
- **벡터 임베딩** = 각 카드/실에 붙인 **의미 태그** (비슷한 의미끼리 가까이 모임)

---

## 3-1단계: 키워드 추출 — "뭘 찾을지 정리하기"

사용자가 **"플랜티넷 2026년 개발 계획 알려줘"** 라고 물었습니다.

LLM이 이 질문을 분석해서 **두 종류의 검색어**를 뽑습니다:

```
                    사용자 질문
                        │
                        ▼
              ┌─────────────────────┐
              │   LLM에게 질문 분석  │
              │   "이 질문에서 핵심   │
              │    단어를 뽑아줘"    │
              └─────────────────────┘
                    │         │
          ┌─────────┘         └──────────┐
          ▼                              ▼
   high_level (상위 개념)         low_level (구체적 이름)
   "개발 계획"                    "플랜티넷"
   "기술 연구소"                  "2026년"
```

### 왜 두 종류로 나누나?

| 종류 | 비유 | 역할 |
|------|------|------|
| **high_level** | "요리 레시피 찾아줘" → **요리** 라는 **주제** | 큰 그림을 잡음 — "어떤 종류의 관계를 찾을까?" |
| **low_level** | "김치찌개 레시피 찾아줘" → **김치찌개** 라는 **구체적 이름** | 정확한 대상을 잡음 — "어떤 엔티티를 찾을까?" |

---

## 3-2단계: Hybrid KG 검색 — "두 가지 방법으로 동시에 찾기"

여기가 핵심입니다. **두 가지 검색을 동시에** 하고, 결과를 합칩니다.

### (A) Local 검색 — "이름으로 카드 찾고, 실 따라가기"

low_level 키워드 `"플랜티넷", "2026년"`으로 **엔티티 카드**를 먼저 찾습니다.

```
검색어: "플랜티넷, 2026년"
         │
         ▼
  ┌──────────────────────────────────────┐
  │  엔티티 벡터DB (938개 카드)           │
  │                                      │
  │  각 카드에는 "의미 태그"(임베딩)가 있음  │
  │  → "플랜티넷"과 의미가 가까운 카드 검색  │
  │                                      │
  │  📌 PLANTYNET ............. 유사도 0.92 │  ← 가장 비슷!
  │  📌 기술연구소 ............ 유사도 0.78 │
  │  📌 2026년 개발계획 ....... 유사도 0.75 │
  │  📌 AI 모델 성능 관리 ..... 유사도 0.61 │
  │  📌 유해 콘텐츠 차단 ...... 유사도 0.45 │
  │  ...                                 │
  └──────────────────────────────────────┘
```

카드를 찾았으면, 그 카드에 연결된 **실(관계)**도 따라갑니다:

```
  PLANTYNET 카드에서 실 따라가기:
  
  PLANTYNET ──소속──→ 기술연구소
  PLANTYNET ──개발──→ 유해 콘텐츠 차단
  PLANTYNET ──계획──→ 2026년 개발계획
            └──사용──→ AI 기술
```

**결과:** "PLANTYNET" 카드 + 연결된 관계 4개

---

### (B) Global 검색 — "관계(실)를 먼저 찾고, 카드 따라가기"

high_level 키워드 `"개발 계획", "기술 연구소"`로 **관계(실)**를 먼저 찾습니다.

```
검색어: "개발 계획, 기술 연구소"
         │
         ▼
  ┌──────────────────────────────────────────────┐
  │  관계 벡터DB (1,281개 실)                     │
  │                                              │
  │  🔗 PLANTYNET → 기술연구소: "소속" .. 유사도 0.88 │  ← 비슷!
  │  🔗 개발계획 → AI과제: "포함" ........ 유사도 0.82 │
  │  🔗 기술연구소 → SNS필터링: "담당" ... 유사도 0.71 │
  │  🔗 XR차단 → 개발계획: "포함" ........ 유사도 0.65 │
  │  ...                                         │
  └──────────────────────────────────────────────┘
```

관계를 찾았으면, 그 관계의 **양쪽 카드(엔티티)**도 수집합니다:

```
  "PLANTYNET → 기술연구소: 소속" 이라는 실을 찾았으면
  → PLANTYNET 카드도 수집
  → 기술연구소 카드도 수집
```

**결과:** 관계 4개 + 양쪽 엔티티들

---

### (C) Round-Robin 병합 — "두 결과를 공평하게 섞기"

Local과 Global 결과를 **번갈아가며** 합칩니다:

```
Local 결과:  [PLANTYNET, 2026년개발계획, 기술연구소, AI모델]
Global 결과: [기술연구소, SNS필터링, XR차단, 개발계획]

병합 (번갈아 가며):
  1. PLANTYNET      (Local)
  2. 기술연구소      (Global) — 중복이면 건너뜀
  3. 2026년개발계획  (Local)
  4. SNS필터링       (Global)
  5. 기술연구소      (Local) — 이미 있으므로 건너뜀
  6. XR차단          (Global)
  7. AI모델          (Local)
  8. 개발계획        (Global)

최종: [PLANTYNET, 기술연구소, 2026년개발계획, SNS필터링, AI모델, XR차단, 개발계획]
```

### 왜 번갈아 섞나?

한쪽 결과만 쓰면 편향됩니다:

- **Local만** → "플랜티넷" 주변만 보고, "개발 계획"이라는 큰 주제를 놓침
- **Global만** → "개발 계획" 관련 관계는 많지만, "플랜티넷"이라는 특정 회사를 놓침
- **번갈아 섞으면** → 두 관점을 균형 있게 포함

## 3-3단계: 토큰 절단 + 청크 병합 — "책 페이지 찾아 붙이기"

지금까지 찾은 건 **색인 카드와 실**뿐입니다. 실제 답변을 만들려면 **원본 텍스트**(책 페이지)가 필요합니다.

### Step 1: 너무 많으면 자르기 (토큰 절단)

```
┌────────────────────────────────────────────────┐
│  찾은 엔티티 30개, 관계 50개                     │
│  → LLM에 한번에 보내기엔 너무 많음               │
│  → 유사도 높은 순서대로 잘라서 상위 N개만 유지    │
│                                                │
│  예) 엔티티 30개 → 상위 10개만                   │
│      관계 50개 → 상위 15개만                     │
└────────────────────────────────────────────────┘
```

### Step 2: 원본 텍스트 청크 연결 (청크 병합)

```
┌────────────────────────────────────────────────┐
│  각 엔티티/관계에는 "어떤 청크에서 왔는지"        │
│  source_id가 기록되어 있음                       │
│                                                │
│  PLANTYNET 엔티티                               │
│    → source_id: chunk-8de6e6c3                  │
│    → 원본 청크: "[Document: 플랜티넷_개발계획.pdf]│
│       2026년 기술연구소 개발 계획..."             │
│                                                │
│  이렇게 관련 청크들을 모아서 "원본 맥락" 복원    │
└────────────────────────────────────────────────┘
```

**비유:** 도서관에서 색인 카드를 찾았으면, 그 카드에 적힌 "3번 서가, 27페이지"를 보고 **실제 책 페이지를 펴서 읽는** 단계입니다.

---

## 3-4단계: 답변 생성 — "모은 정보로 보고서 작성"

지금까지 수집한 **엔티티 + 관계 + 원본 청크**를 하나의 문서("컨텍스트")로 정리하고, LLM에게 "이걸 보고 답변해"라고 합니다.

```
┌─────────────── LLM에게 전달하는 내용 ───────────────┐
│                                                    │
│  시스템: "아래 Context 안의 정보만 사용해서 답변해"   │
│                                                    │
│  -----Context-----                                 │
│                                                    │
│  [엔티티 정보]                                      │
│  • PLANTYNET: 대한민국 IT 기업, 유해콘텐츠 차단 전문  │
│  • 기술연구소: 플랜티넷 산하 연구 조직               │
│  • 2026년 개발계획: 연간 개발 로드맵                 │
│                                                    │
│  [관계 정보]                                        │
│  • PLANTYNET → 기술연구소: 소속                     │
│  • 개발계획 → AI과제: 포함                          │
│  • 개발계획 → XR차단: 포함                          │
│                                                    │
│  [원본 청크]                                        │
│  "2026년 기술연구소 개발 계획에는 XR 디바이스용       │
│   유해 콘텐츠 차단, AI 모델 성능 관리 시스템,        │
│   SNS 환경 필터링 기술 개발이 포함되어 있습니다..."   │
│                                                    │
│  -----End Context-----                             │
│                                                    │
│  사용자 질문: "플랜티넷 2026년 개발 계획 알려줘"     │
│                                                    │
└────────────────────────────────────────────────────┘
                    │
                    ▼
            ┌──────────────┐
            │  LLM이 답변   │
            │  생성         │
            └──────────────┘
                    │
                    ▼
  "플랜티넷 기술연구소의 2026년 개발 계획에는
   1) XR 디바이스용 유해 콘텐츠 차단 앱 개발
   2) AI 모델 성능 관리 시스템 구축
   3) SNS 환경 온디바이스 필터링 기술 개발
   이 포함되어 있습니다."
```

**핵심 제약:** LLM은 **Context 안에 있는 정보만** 사용합니다. 아무리 자기가 아는 게 있어도 Context에 없으면 답변에 넣지 않습니다. → **환각(Hallucination) 방지**

## 전체를 한 장으로 요약

```
"플랜티넷 2026년 개발 계획 알려줘"
         │
   ┌─────┴─────┐
   │ 3-1 키워드  │  LLM이 질문을 분석
   │   추출     │  → 상위 개념: "개발 계획"
   │            │  → 구체적 이름: "플랜티넷"
   └─────┬─────┘
         │
   ┌─────┴──────────────────────────────────┐
   │ 3-2 Hybrid 검색                         │
   │                                        │
   │  Local (이름→카드→실)                   │
   │  "플랜티넷" → 🃏PLANTYNET 카드 발견      │
   │            → 연결된 실 4개 수집          │
   │                                        │
   │  Global (주제→실→카드)                  │
   │  "개발 계획" → 🔗관련 실 4개 발견        │
   │             → 양쪽 카드 수집            │
   │                                        │
   │  병합: 번갈아 섞어서 중복 제거           │
   └─────┬──────────────────────────────────┘
         │
   ┌─────┴──────┐
   │ 3-3 청크    │  카드의 source_id 따라가서
   │   병합     │  원본 "책 페이지" 수집
   └─────┬──────┘
         │
   ┌─────┴──────┐
   │ 3-4 답변    │  엔티티+관계+원본을 하나로 묶어
   │   생성     │  LLM에게 "이걸 보고 답변해"
   └─────┬──────┘
         │
         ▼
   최종 답변 완성
```

## 한 줄 요약

질문에서 키워드를 뽑고 → 두 방향(이름/주제)으로 그래프를 탐색하고 → 원본 텍스트를 찾아 붙이고 → LLM이 그것만 보고 답변합니다.

---

## 전통 RAG와의 핵심 차이

| | 전통 RAG | Graph RAG (LightRAG) |
|---|---|---|
| 검색 단위 | 텍스트 청크만 | 엔티티 + 관계 + 청크 |
| 검색 방식 | 벡터 유사도만 | 벡터 유사도 + 그래프 탐색 |
| 정보 연결 | 없음 (독립 청크) | 엔티티 간 관계로 연결 |
| 다단계 추론 | 어려움 | 관계 체인 따라 가능 |
| 비유 | 책에서 비슷한 페이지만 찾음 | 색인 카드 → 실 → 원본 페이지 순서로 찾음 |
